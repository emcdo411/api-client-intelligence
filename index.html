Primary diagnostic gate

Can the API produce what the lender needs, and can the gap be governed?

The Conard Layer fires before any dashboard data is trusted. It asks three production-readiness questions a Client Integrations BA has to own.

Gate 1 — API contract fidelity

What is the lender's expected field-level schema, latency, and data format?

Risk if unanswered: schema mismatch causes silent data corruption downstream.

Gate 2 — Compliance boundary

Where does RESPA/TILA exposure live in this workflow?

Risk if unanswered: a field-level error is not a bug. It is a compliance event.

Gate 3 — Cycle-time delta

What measurable reduction in title decisioning time does this integration produce?

Risk if unanswered: there is no business case to defend in a lender review.

Compliance posture — Conard Gate 2

RESPA data integrity: 45 of 47 clients are passing MISMO 3.4 validation.

TILA disclosure timing: two legacy XML clients are not triggering the LE disclosure window correctly.

CFPB audit trail: full event log retained per transaction with immutable timestamps.

ML model governance: AI decisioning outputs include confidence scores and human override logs.

Governance gap log — open items

TOR-2206-0178: FISERV legacy XML is missing the required SettlementChargeType node on loan modification workflows.

TOR-2206-0191: Optimal Blue is delivering partial MISMO 3.4 payload with PropertyIdentifier truncated after the first record.

The real question is not whether the integration works. The real question is whether the client can trust what it produces.
