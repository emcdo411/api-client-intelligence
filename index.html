TriggerBounded The agent may only respond to defined LOS event types. Unrecognized event schemas are rejected before the Postman Flows sequence starts. Event payload is logged with timestamp before any action is taken.
Step 1Postman Context Loaded

The active Collection version, environment (sandbox vs production), and OAuth credential set are resolved before the first HTTP call. Version pinning is mandatory — a Collection update requires a new deployment gate, not a live swap.
Postman FlowsVersion pin Collection version is locked at deployment. Environment variables are resolved from the Postman Vault — no credentials in the Collection JSON. If the environment resolves to production and the Collection version is unreviewed, the flow halts.
Step 2MCP Tool Permission Check
Agent confirms it holds permission for every MCP tool required by this order type
The agent checks the MCP Tool Registry before executing. Write-capable tools require CTO-level approval. If any required tool is in pending or blocked state, the flow halts and routes to a named human reviewer.
MCP RegistryWrite gate Auto-approved tools: Postman Context, Schema Validator, Payload Generator, Status Poller. Approval-required tools: Collection Editor, Lender Writer, Compliance Hold Router. A missing approval is not a retry — it is an escalation event with an immutable log entry.
Step 3AI Request Executed
AI model generates or validates the order payload — confidence score is captured before the POST fires
Where AI is used to generate or transform the order payload (field mapping, schema translation, exception pre-scoring), the model output must include a confidence score. Scores below the D1 threshold trigger a hold — the payload does not advance to Step 4 until a human reviewer clears it.
Deepak D1Confidence floor Confidence floor: 0.82. Below floor: payload is flagged, confidence score is written to the audit log, named reviewer is assigned, and SLA timer starts. The AI output is never silently accepted — the rationale tag is required in every production payload.
Step 4Schema Validation
Postman Schema Validator asserts MISMO 3.4 fidelity before the order POST reaches EXOS
The Schema Validator MCP tool runs the full MISMO 3.4 field map against the payload. Required fields, null behaviors, and data type constraints are all asserted. A partial payload is not a warning — it is a fail that routes back to Step 3 with the violation list attached.
Auto-approved toolConard Gate 1 Validated fields: loan identifiers, APN, legal description, borrower PII (encrypted), lender ID, loan amount, closing date. Step 5Conard Gate
Three production-readiness questions must be answerable before the order reaches EXOS fulfillment
Gate 1: schema fidelity confirmed. Gate 2: RESPA/TILA compliance boundary identified. Gate 3: cycle-time baseline and delta confirmed. An order that cannot pass all three gates is not sent to production — it is held with a governance log entry that names the open gate and the responsible owner.
Gate 1 passGate 2 conditionalGate 3 open Current posture: Gate 1 passes for MISMO 3.4 clients. Gate 2 is conditional — TILA LE disclosure window is not confirmed for legacy XML clients. Gate 3 is open — cycle-time baseline has not been formally documented for loan modification workflows.
Step 6Deepak Gate
AI accountability chain is confirmed — explainability, override path, and regulatory trigger map are all live
The Deepak Gate fires on every order where AI has touched the decisioning path. It is not a one-time deployment check — it is a per-transaction governance assertion. If the AI confidence score, rationale tag, or human override log is absent from this transaction, the gate blocks advancement to Step 8.

Step 7Human Override Check
Any COMPLIANCE_HOLD or confidence-floor failure routes to a named human reviewer with an SLA timer
Human-in-the-loop is not a fallback — it is a designed state in the agent lifecycle. The agent is bounded: it cannot resolve a compliance hold autonomously. The reviewer receives the full audit context, the AI rationale tag, the Deepak gate verdict, and the Conard gate status.

The production commit writes the final order state, all intermediate governance verdicts, the AI confidence score, the human override log (if triggered), and the EXOS fulfillment timestamp to the immutable event log. Deepak Governance Engine

AI decisioning in a regulated mortgage API is not a model problem. It is a governance problem.

The Deepak Governance Engine fires after the Conard Gate and before any AI output is trusted by a lender system. It answers five accountability questions that no uptime dashboard can answer.

Governance architecture — DACR v2.6

Five layers. One accountability chain.

EXOS instant decisioning compresses a 72-hour manual title process to 4.2 seconds. That compression only creates lender value when the AI producing the decision can be audited, explained, overridden, and defended to a regulator. The Deepak Engine makes that audit chain visible before a lender relationship is exposed.

◆

D1 — Model Explainability Gate

Can the AI decisioning output be explained to a lender compliance officer in plain language? Every instant title decision must carry a confidence score, a rationale summary, and a human override log. Absent these, the output is not defensible under CFPB examination.

◆

D2 — Bias and Disparate Impact Audit

Does the AI scoring surface introduce disparate impact across borrower demographics? ECOA and HMDA exposure attach when an automated system routes orders or flags exceptions by property type, geography, or loan amount without documented bias controls and regular disparity testing.

◆

D3 — Model Drift and Retraining Protocol

Is the model still accurate against current market conditions? Title exception rates shift with rate cycles, property market dynamics, and servicer behavior. A model trained on 2021 purchase volume has not seen the 2023 to 2025 lock-in environment. Drift detection must be automated and trigger a defined retraining gate.

◆

D4 — Human Override Architecture

Is the human override path operationalized or theoretical? A compliance examiner will ask to see the override log, the named reviewer, the escalation path, and the SLA. If the path exists only in documentation and not in the production order management system, the audit will expose it.

◆

D5 — Regulatory Change Trigger Map

Which regulatory changes would force a model update or a process hold? CO AI Act (June 2026) and CA ADMT (January 2027) introduce automated decision accountability requirements that apply to any AI system making or materially influencing a credit-adjacent decision. ServiceLink must map which EXOS AI outputs fall inside those definitions now.

◆

D6 — Vendor AI Accountability Transfer

When EXOS AI outputs flow into a lender LOS, who owns the decision for regulatory purposes? The Deepak Engine forces a written accountability transfer clause into every integration SOW before the first production API call. Absent that clause, the lender carries the AI governance exposure regardless of who built the model.

Regulatory trigger

CO AI Act — June 2026

Colorado's Artificial Intelligence Act requires covered entities using AI in consequential decisions — defined broadly enough to include automated title exception scoring — to implement governance programs, conduct impact assessments, and provide consumer notice. EXOS clients with CO loan volume must confirm scope before June.

Regulatory trigger

CA ADMT — January 2027

California's Automated Decision Management Technology regulation extends opt-out and impact assessment obligations to any system that makes or materially influences significant decisions. A title commitment is a consequential output in a loan closing chain. ServiceLink's AI governance posture must be documented before CA volume hits the ADMT threshold.

Governance gap

CFPB Model Risk Guidance

CFPB supervisory guidance on model risk management extends SR 11-7 expectations to mortgage-adjacent AI systems. The Deepak Engine maps EXOS AI decisioning outputs against the three SR 11-7 risk categories: conceptual soundness, ongoing monitoring, and outcome analysis. Any gap becomes a finding in a technology service provider examination.

Governance gap

MISMO AI Governance Alignment

MISMO's emerging AI and data governance workgroup is producing standards for AI explainability in mortgage data exchange. Early alignment — documented in the EXOS schema layer — creates a competitive signal to lender partners that ServiceLink is not waiting for enforcement to build governance architecture.

Agentic AI Integration

What Changes When the Agent Is AI

When the orchestration layer is driven by an AI agent rather than a scheduled monitor, three governance requirements activate that do not exist in conventional API integration: the agent must be bounded, observable, and reversible. Bounded means it cannot initiate actions outside the defined Collection scope. Observable means every action produces a timestamped audit record. Reversible means the cancellation and rollback paths are tested before any agentic deployment goes to production.

  • Action boundary enforcement — agent cannot call undocumented endpoints
  • Confidence floor gate — agent pauses and escalates if decisioning score drops below threshold
  • Human-in-the-loop trigger — COMPLIANCE_HOLD state always routes to a named human
  • Rollback test required before production promotion — Postman UAT gate

API architecture brief

The title pull is not one API. It is a two-surface integration system.

This upgrade adds the ServiceLink API BRD explanation: REST for modern order intake and status, SOAP for legacy MISMO/XML title flows, and a Conard Layer gate that decides whether testing can actually start.

REST Order intake + status

Modern LOS platforms send JSON order objects to EXOS using HTTP POST. EXOS returns an order ID, then the client either polls status with GET or receives webhook callbacks. Completed title data returns through status payloads or document retrieval endpoints.

Architecture overview — EXOS title pull system

What EXOS Close actually is

EXOS Close is ServiceLink's closing and title fulfillment platform. When a lender client integrates via API, EXOS becomes the order intake, status management, and fulfillment orchestration layer. The client system does not touch the title directly. It submits an order object to EXOS, EXOS routes to the correct title agent or fulfillment queue, and status updates flow back.

What REST handlesREST

Auth and transportBRD-API-02
Auth pattern:OAuth 2.0 client credentials or API key. mTLS available for enterprise clients.
Transport:TLS 1.2 minimum. PII fields must be encrypted in the payload, never query parameters.
IP whitelist:Client IP range must be submitted before sprint 1. This is a common day-one blocker.
Rate limits:Calls per minute/hour/day must be confirmed; retry logic must use exponential backoff.
