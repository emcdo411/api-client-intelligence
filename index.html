Primary diagnostic gate

Can the API produce what the lender needs, and can the gap be governed?

The Conard Layer fires before any dashboard data is trusted. It asks three production-readiness questions a Client Integrations BA has to own.

Gate 1 — API contract fidelity

What is the lender's expected field-level schema, latency, and data format?

Risk if unanswered: schema mismatch causes silent data corruption downstream.

Gate 2 — Compliance boundary

Where does RESPA/TILA exposure live in this workflow?

Risk if unanswered: a field-level error is not a bug. It is a compliance event.

Gate 3 — Cycle-time delta

What measurable reduction in title decisioning time does this integration produce?

Risk if unanswered: there is no business case to defend in a lender review.

Agentic AI integration gate

Can a technical lead trust what the agent is authorized to do?

The Deepak Layer fires before any artifact involving agentic AI tooling for SOAP/REST lender API integrations -- Postman collection architecture, schema validation design, regression automation, or Claude Code/Claude Skills build patterns. Named for ServiceLink's Technical Lead. Where Conard governs the move from operational signal to client trust, Deepak governs the move from engineering concept to backend implementation.

API architecture brief

The title pull is not one API. It is a two-surface integration system.

This upgrade adds the ServiceLink API BRD explanation: REST for modern order intake and status, SOAP for legacy MISMO/XML title flows, and a Conard Layer gate that decides whether testing can actually start.

REST Order intake + status

Modern LOS platforms send JSON order objects to EXOS using HTTP POST. EXOS returns an order ID, then the client either polls status with GET or receives webhook callbacks. Completed title data returns through status payloads or document retrieval endpoints.

StepActionBRD field
1Client LOS sends order initiation payload to EXOSBRD-SCOPE-02
2EXOS returns 200/201 acknowledgement or 4xx validation errorBRD-SCHEMA-06
3Client polls status or receives webhook callbackBRD-ERR-02
4Title data returned by status payload or document endpointBRD-SCOPE-01

SOAP Legacy schema + MISMO translation

SOAP remains in scope because title and mortgage servicing environments still carry MISMO 2.x and proprietary XML schemas. This layer accepts older order objects, returns title commitment data in XML envelopes, and supports schema translation when the client cannot produce clean JSON.

Architecture overview — EXOS title pull system

What EXOS Close actually is

EXOS Close is ServiceLink's closing and title fulfillment platform. When a lender client integrates via API, EXOS becomes the order intake, status management, and fulfillment orchestration layer. The client system does not touch the title directly. It submits an order object to EXOS, EXOS routes to the correct title agent or fulfillment queue, and status updates flow back.

REST API layer — order intake and status

What REST handlesREST
1Client LOS sends order initiation payload via HTTP POST to EXOS.
2EXOS returns order ID and acknowledgement or validation error.
3Client polls status by order ID or receives webhooks.

Auth and transportBRD-API-02
Auth pattern:OAuth 2.0 client credentials or API key. mTLS available for enterprise clients.
Transport:TLS 1.2 minimum. PII fields must be encrypted in the payload, never query parameters.
IP whitelist:Client IP range must be submitted before sprint 1. This is a common day-one blocker.
Rate limits:Calls per minute/hour/day must be confirmed; retry logic must use exponential backoff.
