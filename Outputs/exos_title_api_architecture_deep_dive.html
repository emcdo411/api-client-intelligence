EXOS Title API architecture: REST and SOAP flows, schema layers, and Conard gate mapping from the ServiceLink API BRD

Architecture overview — EXOS title pull system

What EXOS Close actually is

EXOS Close is ServiceLink's closing and title fulfillment platform. When a lender client integrates via API, EXOS becomes the order intake, status management, and fulfillment orchestration layer. The client system does not touch the title directly — it submits an order object to EXOS, EXOS routes to the correct title agent or fulfillment queue, and status updates flow back. The architecture has two distinct API surfaces: a REST layer for order management and a SOAP layer that handles legacy title data schemas and MISMO translation. Both must be understood and both are in scope in your BRD.

REST API layer — order intake and status

What REST handlesREST
1Client LOS sends order initiation payload via HTTP POST to EXOS order intake endpoint
2EXOS returns an order ID and acknowledgement (200/201) or a validation error (4xx)
3Client polls order status via GET using the order ID, or receives webhook callbacks if configured
4Completed title data is returned in the status payload or via a separate document retrieval endpoint
Auth and transport (REST)BRD-API-02
Auth pattern:OAuth 2.0 client credentials or API key depending on ServiceLink environment provisioning. mTLS available for enterprise clients — requires cert lead time.
Transport:TLS 1.2 minimum. PII fields (SSN, full DOB, borrower name) must be in encrypted payload — not query parameters.
IP whitelist:BRD-API-03 — client IP range must be submitted to ServiceLink API ops before sprint 1. This is the most common day-one blocker.
Rate limits:BRD-API-04 — calls per minute/hour/day must be confirmed. Retry logic must use exponential backoff against this limit.

SOAP API layer — legacy schema and MISMO translation

What SOAP handlesSOAP

The SOAP surface exists because the title industry runs heavily on MISMO 2.x and 3.4 schemas inherited from legacy LOS platforms (ICE Encompass, Empower, Fiserv). EXOS maintains a SOAP interface to:

AAccept MISMO-formatted order objects from older LOS platforms that cannot output JSON
BReturn title commitment data, search results, and exception states in MISMO-compliant XML envelopes
CSupport clients with proprietary schemas via a translation layer that maps non-standard fields to EXOS-accepted values
SOAP vs REST decisionBRD-SCHEMA-02
MISMO 3.4:Use REST with JSON payload. MISMO 3.4-aware LOS platforms (modern Encompass, newer MSP builds) output JSON-compatible field maps.
MISMO 2.x:Use SOAP or REST with a translation layer. Budget 1-2 additional sprints for schema translation build and test.
Proprietary:BRD-SCHEMA-03 required. Custom fields not in the EXOS schema must be designated as pass-through, dropped, or errored before sprint 1.
Mixed:Some clients output MISMO 3.4 for origination but 2.x for servicing. Both surfaces may be in scope simultaneously.

Schema layers — what the title pull actually contains

Order intake payload — mandatory field map (BRD-SCHEMA-04)Required before sprint 1
Primary object:Loan number, property address, borrower name — the three anchors EXOS needs to route the order
Property fields:APN, legal description, county, state, property type, transaction type (purchase/refi)
Borrower fields:Name, SSN (encrypted), address — RESPA-governed, PII in transit rules apply
Lender fields:Lender ID, loan amount, closing date, product type — drives fulfillment routing in EXOS
Null handling:BRD-SCHEMA-05 — does a null required field return 4xx or silently drop? Must be tested both ways before UAT
Custom fields:BRD-SCHEMA-03 — any field not in the EXOS schema must have an explicit handling decision before dev writes the mapper
Title pull response payload — what EXOS sends back (BRD-SCHEMA-06)Status code map required
Order received:Order ID + acknowledgement — triggers client-side order tracking record creation
In fulfillment:Intermediate status — client must not timeout waiting; async polling or webhook required
Exception / hold:Compliance hold, title defect, missing doc — triggers manual review workflow; BRD-ERR-04 recipient must be named
Title commitment:Document URL or embedded XML — the actual title data. Triggers closing workflow on client side
Completed:Order closed and archived — triggers final status update in client LOS
Cancelled:BRD-ERR-05 — cancellation must have a dedicated endpoint. RESPA requires a documented cancellation path before go-live
